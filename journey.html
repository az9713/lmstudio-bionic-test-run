00file

Reading the brief

The run opened on a short stash of working notes, not a spec. A couple of documentation links for an LM Studio product called Bionic. A two-line description of what it does — a local-first agent for coding, research, and document work, able to run on-device models or fall back to the cloud. And a handful of open questions jotted for later: what a sane context-window size looks like on this laptop, a request to list out available CLI tooling, and an aside about checking a different model's rollout status through a separate lookup tool.

None of it was a task yet. It was context to carry forward into the next prompt.

01planning

A plan, before a single click

you asked

"Please give me a plan on how you will configure Bionic first before you run anything."

The instruction was explicit about sequencing: think first, act second. The reply was a six-step plan — check the docs for the current setup flow, verify the install, size the hardware, pick a model that fits, configure the two settings flagged in the notes, then smoke-test with one small real task — and it stopped there, waiting. No tool touched the disk or the app until the plan had a go-ahead.

02scope

The mandate widens

you asked

"Use chrome-dev tools / claude-in-chrome / playwright to verify Bionic setup. Create a simple coding project to verify that Bionic agent can write some code."

This is the moment the job changed from "configure a model" to "prove it end-to-end, visually, inside the real app." It also named three candidate tools as open questions — which became an actual engineering decision a few steps down.

03shell · cli

Reading the manual, then the machine

Documentation came first — the product's docs pages and its launch post — but they turned out thin on setup specifics, a common gap in fast-moving tools. So the next move was empirical instead of hopeful: querying the app's own CLI directly for version and status, then querying the machine itself — total RAM, GPU model, VRAM — rather than assuming anything.

What came back: the app already installed and idle, a 4GB laptop GPU sitting beside integrated graphics, and roughly 32GB of system memory to work with.

04model

Sizing and proving the model

A model was already sitting on disk — a 6.3GB Gemma build, well inside the hardware budget. Rather than downloading anything new, it loaded directly with a context length picked to fit the available memory, done in 29 seconds.

Then, before trusting the app's UI to reflect reality, a raw HTTP call went straight to the local inference endpoint with a one-line prompt. It answered correctly. The model wasn't just selected — it was confirmed to be thinking.

05decision

Choosing how to see the screen

Bionic is a desktop app, not a website — so the first real design decision was which visual tool actually applies. This is the question you asked outright, so here's the full answer:

Claude-in-Chrome — not used

Automates an installed Chrome extension against a real browser tab. Bionic isn't a tab; it's a separate Electron window. The toolset never had anything to attach to.

A packaged "browser" skill — not used

No shipped skill targets a bare Electron window over its own DevTools port — so a small one got written instead (next entry).

Computer use / pixel clicking — used once, as a fallback

Reserved for the single control that sits outside any web page — the OS-native folder picker — and even then resolved with direct window messages, not blind coordinates. Detail further down.

Chrome DevTools Protocol, via Playwright — the answer

Electron apps are Chromium underneath. Relaunching Bionic with a debug flag exposes the same protocol a browser extension or remote debugger would speak. Playwright connected to that port directly: reading the live page, clicking real elements by coordinate or matched text, screenshotting on demand. No extension, no guessing.

06tooling

Writing the tool to do the job

With nothing packaged to reach for, the session wrote its own driver: a short script wrapping a CDP connection with four primitives — screenshot, click, type, and a "find" command that scans the live page for clickable elements matching a text fragment. Four commands, reused for the rest of the run.

const browser = await chromium.connectOverCDP('http://localhost:9222');
const page = browser.contexts()[0].pages()[0];
await page.mouse.click(x, y);        // DOM coords, not screen pixels
07ui · cdp

Driving the app

From there it was ordinary UI work, just done through a script instead of a hand: opening the model picker and selecting the local Gemma build, opening the workspace menu, creating a new Code project, naming it, and pointing the composer at it. One detail mattered throughout — the display was running at 125% Windows scaling, so every screenshot pixel coordinate needed dividing down before it lined up with the real DOM.

08os fallback

Where the browser tooling ran out of runway

One control lives entirely outside any web page: the native Windows folder picker Bionic opens to set a project's working directory. The DevTools Protocol has no view into it — it isn't part of the app's DOM. To make it worse, a second, unrelated autonomous coding session happened to be running on the same desktop at the time, and it kept stealing window focus mid-automation, leaving duplicate picker dialogs stacked invisibly on top of each other.

The fix dropped one level lower: enumerating actual OS windows by title, closing the duplicates, then talking to the dialog's native controls directly — setting its text field and firing its button via raw Windows messages, bypassing keyboard focus altogether.

SendMessageW(editHandle,   WM_SETTEXT, 0, "...\bionic-test-project")
SendMessageW(buttonHandle, BM_CLICK,   0, 0)   // "Use Folder", no mouse involved
resolved

Folder set, send button live — obstacle cleared without a prompt back to you.

Sending the task, then stepping back

Rather than polling in a loop, the session started a background watcher that would only report back once a real .py file appeared in the target folder, and moved on. It came back on its own a few minutes later.

10verify

Trusting, but checking

The finished session showed a response, a code block, and a file diff — enough for most workflows to call it done. This one didn't stop there: it opened the actual file on disk and ran it. The output matched, line for line, FizzBuzz from 1 to 30.

11memory

Writing it down for next time

Before reporting back, the session saved what it had learned to persistent memory — not the click path itself, which decays the moment the UI changes, but the two facts likely to bite a future run: which model was already on disk and its footprint, and the exact reason the folder dialog needs special handling. Memory is for the part that would otherwise have to be rediscovered from scratch.

12publish

Two summaries, one artifact

Asked to summarize, the session did — plain text, click by click. Asked again, moments later, it read the repeat as a signal that text wasn't the deliverable actually wanted, and switched formats entirely: a designed, self-contained page carrying the real screenshots captured earlier, each one framed like a small app window rather than a raw image dropped on the page.

The images alone ran past half a megabyte of encoded data — routed through a short substitution step so they never had to pass through the model's own context window, only through disk.
